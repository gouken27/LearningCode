## 1. Что такое HTML и что он делает в браузере

### Три языка веба и их роли

Веб-страница — это всегда три отдельных слоя, и каждый делает своё дело:

```
┌─────────────────────────────────────────────────────────────┐
│                    ТРИ СЛОЯ ВЕБ-СТРАНИЦЫ                    │
│                                                             │
│  HTML                CSS               JavaScript           │
│  ──────              ───               ──────────           │
│  Структура           Внешний вид        Поведение            │
│  и смысл             и стили            и логика             │
│                                                             │
│  «Это заголовок»     «Заголовок         «При клике на        │
│  «Это кнопка»         красного цвета,   кнопку — открыть    │
│  «Это список»         24px, жирный»     модальное окно»      │
│                                                             │
│  Работает без CSS    Не работает без    Не работает без      │
│  и без JS            HTML               HTML                 │
└─────────────────────────────────────────────────────────────┘
```

**HTML (HyperText Markup Language)** — это язык разметки, не язык программирования. В нём нет циклов, переменных, условий. Он просто описывает **структуру и смысл** контента с помощью тегов.

### HTML — это разметка смысла, не вёрстка

Главное заблуждение новичков: думать, что HTML — это про «как выглядит». Нет.

```html
<!-- ❌ Так думают новички: HTML — для внешнего вида -->
<div class="big-red-text">Добро пожаловать</div>
<div class="smaller-text">О нас</div>

<!-- ✅ Так правильно: HTML — для смысла, CSS — для вида -->
<h1>Добро пожаловать</h1>
<h2>О нас</h2>
```

Оба варианта могут выглядеть одинаково в браузере. Но второй вариант:
- понятен поисковому роботу (Google знает: это заголовок страницы)
- понятен скринридеру (слепой пользователь слышит «заголовок первого уровня»)
- понятен будущему вам через полгода

### Что такое тег, элемент, атрибут

```
  <a   href="https://example.com"   target="_blank">  Нажми  </a>
  ─┬─  ──────────────────────────   ──────────────    ─────   ───
   │   └─────────────┬─────────────────────────┘      │       │
   │             атрибуты                           контент  закрывающий
   │             (имя="значение")                            тег
   │
   открывающий тег

  └─────────────────────────────────────────────────────────┘
                      элемент целиком
```

- **Тег** — `<a>` или `</a>` — угловые скобки с именем
- **Атрибут** — дополнительная информация внутри открывающего тега (`href`, `target`, `class`, `id`)
- **Элемент** — весь блок от открывающего до закрывающего тега включая содержимое

Некоторые элементы самозакрывающиеся — у них нет содержимого и закрывающего тега:
```html
<img src="cat.jpg" alt="Кот">
<input type="text">
<br>
<hr>
<meta charset="UTF-8">
```

## 2. Как браузер читает HTML: критический путь рендеринга

Это самый важный раздел для понимания того, **почему** многие правила HTML существуют.

### Шаг за шагом: от файла до пикселей

Вы набрали URL и нажали Enter. Что происходит:

```
┌──────────────────────────────────────────────────────────────────┐
│              КРИТИЧЕСКИЙ ПУТЬ РЕНДЕРИНГА                         │
│                                                                  │
│  1. HTML-файл       2. Парсинг          3. Построение DOM        │
│  ─────────────      ─────────────       ─────────────────        │
│  Браузер получает   Браузер читает      Дерево объектов —        │
│  байты по сети,     HTML сверху вниз,   JavaScript-представление │
│  декодирует в       строит токены       HTML-документа           │
│  символы            (теги, атрибуты,                             │
│                     текст)                                       │
│                                                                  │
│  4. Загрузка CSS    5. Построение       6. Render Tree           │
│  ─────────────      CSSOM               ─────────────            │
│  Браузер нашёл      ─────────────       Объединение DOM          │
│  <link rel=         CSS тоже строит     и CSSOM. Только          │
│  "stylesheet">      дерево правил       видимые элементы         │
│  — СТОП, грузим     (CSSOM). Без него   (display:none            │
│                     рендер невозможен   — не входит)             │
│                                                                  │
│  7. Layout          8. Paint            9. Composite             │
│  ─────────          ─────────           ─────────────            │
│  Вычислить          Нарисовать          Слои склеить             │
│  размеры и          пиксели на          в финальную              │
│  положение          экране              картинку                 │
│  каждого элемента                                                │
└──────────────────────────────────────────────────────────────────┘
```

### Почему это важно: три практических следствия

**Следствие 1: CSS блокирует рендеринг**

Браузер не рисует страницу, пока не загрузил и не обработал весь CSS. Потому что без CSSOM неизвестно, как выглядит каждый элемент. Поэтому `<link rel="stylesheet">` ставят в `<head>` — пусть грузится параллельно с разбором HTML, а не после.

**Следствие 2: JS по умолчанию блокирует парсинг HTML**

Когда браузер встречает `<script>`, он останавливает чтение HTML и выполняет скрипт. Почему? Потому что JS может вызвать `document.write()` и изменить HTML прямо в процессе парсинга. Поэтому правило:

```html
<!-- ❌ Плохо: скрипт останавливает парсинг, DOM ещё не готов -->
<head>
  <script src="app.js"></script>
</head>

<!-- ✅ Хорошо: ставим перед </body>, DOM уже построен -->
<body>
  <!-- весь контент -->
  <script src="app.js"></script>
</body>

<!-- ✅ Ещё лучше: defer — грузится параллельно, выполняется после DOM -->
<head>
  <script src="app.js" defer></script>
</head>
```

**Следствие 3: разница между `defer` и `async`**

```
  Без атрибута:
  HTML:   ──────────[СТОП]──────────────────────► дальше
  JS:               ↕ грузим и выполняем

  async:
  HTML:   ──────────────────[СТОП]──────────────► дальше
  JS:         ↕ грузим...     ↕ выполняем (как только загрузился)
  Проблема: порядок выполнения нескольких скриптов непредсказуем

  defer:
  HTML:   ─────────────────────────────────────► дальше
  JS:         ↕ грузим параллельно...            ↕ выполняем после DOM
  Порядок сохраняется. Почти всегда правильный выбор.
```

Правило простое: **почти всегда используйте `defer`** для скриптов в `<head>`.

### Почему React вставляет всё в один `<div id="root">`

Теперь это понятно. React работает через JavaScript. JS строит DOM **не из HTML**, а программно: создаёт элементы, вставляет их, удаляет, перемещает. HTML при этом содержит только одну точку входа — пустой `<div>`, в который React «монтирует» весь интерфейс после загрузки JS.

```html
<!-- Так выглядит HTML React-приложения -->
<body>
  <div id="root"></div>          <!-- пустой контейнер -->
  <script src="bundle.js" defer></script>  <!-- React внутри -->
</body>
<!-- Браузер получает пустую страницу, потом JS всё рисует -->
```

Это и есть причина проблем SPA с SEO — поисковый робот видит пустой `<div>`. Но об этом — в части 4.

## 3. Анатомия HTML-документа

### Минимальный правильный HTML-документ

```html
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Название страницы</title>
</head>
<body>
  <!-- Весь видимый контент -->
</body>
</html>
```

Разберём каждую строку:

```
<!DOCTYPE html>
────────────────
Не тег, а объявление. Говорит браузеру: «это современный HTML».
Без него браузер включает «quirks mode» — режим совместимости
со старыми сайтами 1990-х. Поведение box model и других вещей
меняется непредсказуемо. Всегда первой строкой.

<html lang="ru">
─────────────────
lang — атрибут языка документа. Нужен:
• скринридерам (выбрать правильное произношение)
• браузеру для автоперевода
• поисковикам
Значения: "ru", "en", "ja", "zh" и т.д.

<meta charset="UTF-8">
───────────────────────
UTF-8 — кодировка, которая поддерживает все языки мира,
включая кириллицу, японский, арабский, эмодзи.
Без этого тега кириллица превратится в кракозябры.
ВСЕГДА ПЕРВОЙ СТРОКОЙ В <head> — до любых других тегов.

<meta name="viewport" content="width=device-width, initial-scale=1.0">
───────────────────────────────────────────────────────────────────────
width=device-width:  ширина страницы = ширина экрана устройства
initial-scale=1.0:  начальный масштаб — 1:1, без зума

Без этого мета-тега мобильный браузер рендерит страницу
как будто это десктоп (обычно 980px), потом уменьшает.
Текст становится нечитаемым. Media queries не работают.
```

### Что идёт в `<head>`, что в `<body>`

```
<head> — метаинформация, не видна пользователю:
  ✓ <meta> — кодировка, viewport, описание, og-теги
  ✓ <title> — заголовок вкладки
  ✓ <link> — подключение CSS, шрифтов, иконок
  ✓ <script defer> — можно, если нужно раннее выполнение
  ✗ Видимый контент

<body> — всё, что видит пользователь:
  ✓ Текст, картинки, формы, кнопки
  ✓ <script> в конце — legacy-способ
  ✗ <meta>, <title>, <link rel="stylesheet">
```

## 4. Семантическая разметка

### Каркас типичной страницы

```html
<body>
  <header>
    <!-- шапка сайта: логотип, главная навигация -->
    <nav aria-label="Главная навигация">
      <ul>
        <li><a href="/">Главная</a></li>
        <li><a href="/catalog">Каталог</a></li>
        <li><a href="/music">Музыка</a></li>
      </ul>
    </nav>
  </header>

  <main>
    <!-- основной контент страницы, ОДИН на страницу -->
    <article>
      <!-- самодостаточный контент: пост, карточка, обзор -->
      <header>
        <h1>Название статьи</h1>
        <p>Автор · Дата</p>
      </header>

      <section>
        <!-- тематический раздел С заголовком -->
        <h2>Раздел первый</h2>
        <p>Текст...</p>
      </section>

      <footer>
        <!-- метаданные статьи: теги, ссылки -->
      </footer>
    </article>

    <aside>
      <!-- боковой контент: похожее, реклама, связанное -->
    </aside>
  </main>

  <footer>
    <!-- подвал сайта: копирайт, контакты, ссылки -->
  </footer>
</body>
```

### Шпаргалка: какой тег для чего

```
┌──────────────┬────────────────────────────────┬─────────────────────────────────┐
│  Тег         │  Когда БРАТЬ                    │  Когда НЕ БРАТЬ                 │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <article>    │ Самодостаточный контент:        │ Просто блок текста, который     │
│              │ пост, карточка аниме, коммент   │ не имеет смысла вне страницы    │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <section>    │ Тематический раздел             │ Обёртка для стилей — бери <div> │
│              │ с заголовком внутри             │ Нет заголовка — нет <section>   │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <nav>        │ Главная навигация сайта,        │ Любой список ссылок             │
│              │ хлебные крошки                  │ (ссылки в карточке — не nav)    │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <aside>      │ Сайдбар, похожие записи,        │ Любая боковая колонка без       │
│              │ реклама, биография автора       │ смысловой связи с контентом     │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <main>       │ Основной уникальный контент     │ Шапка, футер, навигация.        │
│              │ страницы. ОДИН на страницу      │ Никогда не вкладывается в aside │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <header>     │ Шапка сайта ИЛИ шапка article/ │ Просто верхний блок без смысла  │
│              │ section (оба варианта верны!)   │                                 │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <footer>     │ Подвал сайта ИЛИ низ article   │ Просто нижний блок без смысла   │
├──────────────┼────────────────────────────────┼─────────────────────────────────┤
│ <div>        │ Обёртка для стилей и layout     │ Когда есть подходящий           │
│              │ без семантического смысла       │ семантический тег               │
└──────────────┴────────────────────────────────┴─────────────────────────────────┘
```

> **Правило большого пальца**: не можешь объяснить одной фразой *почему* взял `<section>` вместо `<div>` — бери `<div>`. Неправильная семантика хуже её отсутствия.

### Заголовки: иерархия важнее размера

```html
<!-- ❌ Плохо: выбираем размер, а не уровень -->
<h1>Anime Catalog</h1>
<h3>Attack on Titan</h3>    <!-- пропущен h2! -->
<h5>Сезон 1</h5>            <!-- пропущен h4! -->

<!-- ✅ Хорошо: строгая иерархия -->
<h1>Anime Catalog</h1>
  <h2>Attack on Titan</h2>
    <h3>Сезон 1</h3>
    <h3>Сезон 2</h3>
  <h2>Demon Slayer</h2>
    <h3>Сезон 1</h3>
```

Если нужен визуально маленький заголовок — пишите правильный уровень (`<h4>`), а размер задайте через CSS. Скринридеры строят оглавление по этой иерархии, как Word строит структуру документа.

### `<strong>`, `<em>` — смысл, а не стиль

```html
<!-- <strong> — важность, акцент -->
<p>Никогда не отправляй пароль через <strong>GET</strong>.</p>

<!-- <em> — выделение голосом, интонация -->
<p>Я сказал <em>почти</em> всегда, не всегда.</p>

<!-- <b> и <i> — визуальное выделение без семантики -->
<p>Название аниме пишется курсивом: <i>Shingeki no Kyojin</i></p>
```

## 5. Формы

### Почему формы сложные

Форма — это место, где HTML напрямую взаимодействует с сервером. Здесь сосредоточена большая часть UX-ошибок: «не работает на мобиле», «пароль улетел в URL», «поле подсвечивается красным до того, как пользователь что-то ввёл».

### Правильная структура формы

```html
<form action="/api/login" method="post" novalidate>

  <!-- label + input всегда вместе -->
  <div class="field">
    <label for="email">Email</label>
    <input
      type="email"
      id="email"
      name="email"
      autocomplete="email"
      required
      placeholder="ivan@example.com"
    >
  </div>

  <div class="field">
    <label for="password">Пароль</label>
    <input
      type="password"
      id="password"
      name="password"
      autocomplete="current-password"
      required
      minlength="8"
    >
    <!-- Подсказка: связана через aria-describedby -->
    <p id="password-hint" class="hint">Минимум 8 символов</p>
  </div>

  <button type="submit">Войти</button>

</form>
```

### Анатомия каждого атрибута

```
type="email"
────────────
Это не просто косметика. Браузер:
• на мобиле показывает клавиатуру с символом @
• проверяет формат (должен быть символ @ и домен)
• подключает менеджер паролей

name="email"
────────────
КЛЮЧ, под которым значение поля полетит на сервер.
POST-запрос будет содержать: email=ivan@example.com&password=...
Если name нет — поле не отправится. id ≠ name.

id="email"
──────────
Только для связи с <label for="email"> и CSS.
Уникален на странице. К серверу отношения не имеет.

autocomplete="email"
────────────────────
Подсказка браузеру для автозаполнения из менеджера паролей.
Без этого браузер угадывает по name/id и часто ошибается.
autocomplete="new-password"  — для поля регистрации
autocomplete="current-password"  — для поля входа
autocomplete="one-time-code"  — для SMS-кода

required
────────
Браузер не пустит форму без этого поля.
НО: это UX-помощник, не безопасность.
Пользователь может отключить JS, отправить curl, поменять HTML.
Валидация ВСЕГДА дублируется на сервере.
```

### Типы полей и когда что выбирать

```html
<!-- Текстовые поля -->
<input type="text">          <!-- произвольный текст -->
<input type="email">         <!-- email + мобильная клавиатура с @ -->
<input type="password">      <!-- скрывает символы -->
<input type="search">        <!-- кнопка очистки × -->
<input type="url">           <!-- валидация URL -->
<input type="tel">           <!-- телефон + цифровая клавиатура -->

<!-- Числа и даты -->
<input type="number">        <!-- только числа + стрелочки -->
<input type="range">         <!-- слайдер -->
<input type="date">          <!-- нативный датапикер -->
<input type="time">
<input type="color">         <!-- нативный colorpicker -->

<!-- Выборы -->
<input type="checkbox">      <!-- ноль или несколько вариантов -->
<input type="radio">         <!-- ровно один вариант -->
<select>                     <!-- выпадающий список -->
  <option value="action">Экшн</option>
  <option value="drama">Драма</option>
</select>

<!-- Многострочный текст -->
<textarea rows="4"></textarea>

<!-- Файлы -->
<input type="file" accept="image/*" multiple>

<!-- Скрытое поле — уходит на сервер, пользователь не видит -->
<input type="hidden" name="csrf_token" value="abc123">
```

> **Ловушка**: не используйте `type="number"` для телефонов, PIN-кодов, индексов. Он выкидывает ведущие нули, ломается со знаком `+`. Используйте `type="tel"` + `inputmode="numeric"` + `pattern="[0-9]*"`.

### GET vs POST: когда что выбирать

```
GET                                POST
───────────────────                ───────────────────────
Данные в URL:                      Данные в теле запроса:
/search?query=aot&genre=action     (не видно в адресной строке)

✓ Можно сохранить в закладку      ✓ Безопасно для паролей
✓ Можно поделиться ссылкой        ✓ Нет ограничения длины
✓ Кешируется браузером            ✓ Для изменения данных

Когда брать:                       Когда брать:
• Поиск                            • Логин / регистрация
• Фильтрация каталога              • Создание/изменение данных
• Пагинация                        • Загрузка файлов
• Любые «чтения»                   • Любые «записи»
```

### Частые ошибки с формами

```html
<!-- ❌ Кнопка без type сабмитит форму -->
<form>
  <input type="text" name="search">
  <button onclick="showPassword()">👁</button>  <!-- сабмит! -->
</form>

<!-- ✅ type="button" для не-сабмит кнопок -->
<form>
  <input type="text" name="search">
  <button type="button" onclick="showPassword()">👁</button>
</form>

<!-- ❌ placeholder вместо label — теряет смысл при вводе -->
<input type="text" placeholder="Введите email">

<!-- ✅ label всегда, placeholder опционально -->
<label for="email">Email</label>
<input type="text" id="email" placeholder="ivan@example.com">

<!-- ❌ disabled поле не отправится на сервер -->
<input type="text" name="username" value="ivan" disabled>

<!-- ✅ readonly — нельзя редактировать, но отправляется -->
<input type="text" name="username" value="ivan" readonly>
```

### Группировка полей: `<fieldset>` и `<legend>`

```html
<!-- Для радио-кнопок обязательно -->
<fieldset>
  <legend>Статус аниме</legend>
  <label>
    <input type="radio" name="status" value="all" checked> Все
  </label>
  <label>
    <input type="radio" name="status" value="ongoing"> Выходит
  </label>
  <label>
```

Скринридер прочтёт: «Статус аниме, группа. Все, выбрано. Выходит. Завершено.» — контекст понятен.

### Картинки

```html

  height="450"

<!-- Картинка с подписью -->

  <img src="/assets/img/covers/fma.jpg" alt="" width="300" height="450">
  <figcaption>Fullmetal Alchemist: Brotherhood — одно из лучших аниме всех времён</figcaption>
```

```
width и height — ОБЯЗАТЕЛЬНЫ даже если CSS задаёт другой размер.
─────────────────────────────────────────────────────────────────
Браузер резервирует место ДО загрузки картинки.
Без этих атрибутов страница «прыгает» когда картинка загружается —
это называется Cumulative Layout Shift (CLS).
CLS влияет на SEO и воспринимается как баг.

loading="lazy"
──────────────
Картинка грузится только когда пользователь скроллит до неё.
Ускоряет первую загрузку страницы.
НЕ ставьте на картинки выше «линии сгиба» (hero-изображения,
логотип) — они должны грузиться сразу.
```

## 7. Доступность (a11y)

### Кто такие пользователи с ограниченными возможностями

Прежде чем пропустить этот раздел — статистика. По данным ВОЗ, около 15% населения Земли имеет ту или иную форму инвалидности. Но доступность помогает шире:

```
Слабовидящие / незрячие       → скринридеры (NVDA, VoiceOver, TalkBack)
Слабовидящие                  → увеличивают шрифт до 200–400%
Дальтоники (8% мужчин)        → не различают красный и зелёный
Двигательные нарушения        → только клавиатура, без мыши
Эпилептики                    → страдают от мигающих анимаций
Пользователи в метро          → экран на солнце, без звука
Пользователи со сломанной рукой → временно без мыши
```

Хорошая доступность = хороший UX для всех.

### Правило 1: правильные теги дают 80% доступности бесплатно

```html
<!-- ❌ Нет ничего: нет фокуса, нет роли, нет клавиатуры -->
<div onclick="openMenu()">Меню</div>

<!-- ✅ Всё есть: фокус, Enter/Space, роль "кнопка" для скринридера -->
<button onclick="openMenu()">Меню</button>
```

`<button>` из коробки умеет:
- получать фокус с Tab
- срабатывать на Enter и Space
- объявляться скринридером как «кнопка»
- показывать hover/focus-стили

Когда вы пишете `<div onclick>`, вы сознательно ломаете всё это.

### Клавиатурная навигация — базовый тест

Откройте свой сайт, уберите мышь и пройдите только по Tab. Проверьте:

```
1. Фокус виден                 — никогда не пишите outline: none без замены
2. Логичный порядок Tab        — следует порядку DOM, а не CSS-порядку
3. Все интерактивные элементы  — кнопки, ссылки, поля доступны
4. Из модалок можно выйти Esc  — базовое ожидание пользователей
5. Фокус не убегает из модалки — Tab крутится внутри (focus trap)
```

```css
/* Минимум для видимого фокуса: */
:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 3px;
}
/* :focus-visible — только при клавиатурной навигации, не при клике мышью */
```

### ARIA — когда семантики не хватает

**Первое правило ARIA: не используй ARIA.** Сначала спроси: есть ли нативный HTML-элемент? Чаще всего — да.

```html
<!-- Кнопка-иконка без видимого текста -->
<button aria-label="Закрыть меню">
  <svg>...</svg>
</button>

<!-- Кнопка, управляющая разворачиваемым блоком -->
<button
  aria-expanded="false"
  aria-controls="filter-panel"
>
  Фильтры
</button>
<div id="filter-panel" hidden>...</div>

<!-- Связь поля с подсказкой -->
<input type="email" aria-describedby="email-hint">
<p id="email-hint">Будет использоваться для входа в аккаунт</p>

<!-- Динамические уведомления -->
<div aria-live="polite" aria-atomic="true">
  <!-- Сюда JS вставляет: "Фильтр применён: Экшн (24 результата)" -->
</div>

<!-- Скрыть декоративную иконку от скринридера -->
<svg aria-hidden="true" focusable="false">...</svg>
```

### Альтернативный текст картинок

```html
<!-- Содержательная: опишите что на картинке -->
<img src="aot-cover.jpg" alt="Attack on Titan — постер первого сезона">

<!-- Декоративная: ПУСТОЙ alt, не отсутствующий -->
<img src="divider.svg" alt="">
<!-- Без alt="" скринридер прочитает имя файла: "divider.svg" -->

<!-- Картинка с подписью — alt пустой, подпись несёт смысл -->
<figure>
  <img src="eren-titan.jpg" alt="">
  <figcaption>Эрен в форме Атакующего Титана, сезон 4</figcaption>
</figure>
```

### Контрастность

По стандарту WCAG AA минимальное соотношение контраста:
- обычный текст: **4.5:1**
- крупный текст (18pt+): **3:1**

Светло-серый текст на белом фоне (`#999` на `#fff` = 2.85:1) не проходит. Проверяйте в DevTools — Chrome показывает контраст в инспекторе цвета.

## 9. `<template>` и `data-*` — предки JSX

### `<template>`: фрагмент HTML без рендеринга

`<template>` — это контейнер для HTML, который браузер **парсит, но не рендерит**. Скрипты внутри не выполняются, картинки не загружаются. Содержимое доступно через JavaScript.

```html
<!-- В HTML: шаблон карточки аниме -->
<template id="franchise-card-template">
  <article class="card">
    <img class="card__cover" src="" alt="">
    <div class="card__body">
      <h2 class="card__title"></h2>
      <p class="card__description"></p>
      <div class="card__genres"></div>
      <span class="card__rating"></span>
    </div>
  </article>
</template>

<!-- В JavaScript: клонируем и заполняем данными -->
<script>
  const template = document.getElementById('franchise-card-template');
  const clone = template.content.cloneNode(true);

  clone.querySelector('.card__title').textContent = 'Attack on Titan';
  clone.querySelector('.card__rating').textContent = '9.1';

  document.querySelector('.catalog').appendChild(clone);
</script>
```

**Почему это важно для понимания React?** JSX — это синтаксический сахар над тем же самым: «описать структуру HTML, заполнить данными, вставить в DOM». React просто делает это автоматически при изменении данных.

### `data-*`: произвольные данные на элементах

Любой атрибут вида `data-что-угодно` — это способ хранить данные прямо в HTML-элементе. JavaScript читает их через `dataset`.

```html
<!-- В HTML: карточки с данными -->
<article
  class="card"
  data-id="aot"
  data-rating="9.1"
  data-genres="action,drama,dark-fantasy"
  data-status="completed"
>
  <h2>Attack on Titan</h2>
  ...
</article>

<!-- В JavaScript: читаем данные -->
<script>
  const card = document.querySelector('.card');

  console.log(card.dataset.id);      // "aot"
  console.log(card.dataset.rating);  // "9.1"
  console.log(card.dataset.genres);  // "action,drama,dark-fantasy"

  // Удобно для фильтрации:
  const genres = card.dataset.genres.split(',');
  // ["action", "drama", "dark-fantasy"]

  // Можно изменять:
  card.dataset.status = 'favorite';
</script>
```

**Когда использовать `data-*`:**
- хранить ID записи на карточке для передачи в JS
- хранить состояние элемента (открыт/закрыт)
- атрибуты для CSS-селекторов (`[data-status="ongoing"]`)
- данные для аналитики без id

## 10. DevTools: вкладка Elements

DevTools — первый инструмент для отладки HTML. Открыть: F12 (или Cmd+Option+I на Mac).

### Что делать во вкладке Elements

```
Дерево DOM слева:
─────────────────
• Нажмите на любой элемент — он подсветится на странице
• Правая кнопка на странице → «Просмотр кода» — переход к элементу в DOM

Панель справа:
──────────────
• Styles — все CSS-правила применённые к элементу
• Computed — итоговые вычисленные значения
• Accessibility — роль, состояние для скринридера

Горячие клавиши:
────────────────
```

- **State** — expanded/collapsed, checked/unchecked

Если у кнопки `Role: generic` — значит там `<div>`, а не `<button>`.

## 11. Что HTML-разработчику делать не нужно

Понимание границ ответственности экономит время и предотвращает плохие решения.

```
ЧТО ДЕЛАЕТ РАЗРАБОТЧИК:              ЧТО ДЕЛАЕТ КТО-ТО ДРУГОЙ:
────────────────────────              ──────────────────────────
Семантическая разметка          ←→    Дизайн макетов (дизайнер)
Структура документа             ←→    Выбор шрифтов, цветов (дизайнер)
Доступность разметки            ←→    UX-исследования (UX-специалист)
SEO-метатеги                    ←→    Контент-стратегия (контент-менеджер)
Формы: структура и атрибуты     ←→    Серверная валидация (бэкенд)
HTML-шаблоны                    ←→    Серверный рендеринг (бэкенд/DevOps)

ЧТО НЕ НУЖНО ДЕЛАТЬ В HTML:
───────────────────────────────────────────────────────────────
× Делать отступы через <br><br><br>   — это CSS margin
× Делать колонки через <table>        — это CSS Grid/Flexbox
× Inline-стили style="color:red"      — это CSS файл
× Жёстко задавать размер картинки     — это CSS
× Создавать «красивые» теги           — это CSS классы
```

```
✓ Объяснить критический путь рендеринга и почему defer лучше

✓ Базовая доступность: фокус, aria-label, aria-live

✓ Работать с вкладкой Elements в DevTools
✓ Написать разметку всех страниц каталога без CSS
```
