## Определения аббревиатур

### VCS
**Version Control System** — система контроля версий.  
Программа, которая отслеживает изменения в наборе файлов с течением времени, позволяя возвращаться к предыдущим состояниям.  
→ *Примеры: Git, Mercurial, SVN, Perforce.*

---

### DVCS
**Distributed Version Control System** — распределённая система контроля версий.  
Каждый участник проекта имеет **полную копию** репозитория, включая всю историю. Работа возможна без постоянного подключения к серверу.  
→ *Git — это DVCS.*

---

### CVCS
**Centralized Version Control System** — централизованная система контроля версий.  
Существует **один центральный сервер**, который хранит всю историю. Участники подключаются к нему для получения и сохранения изменений.  
→ *Примеры: SVN (Subversion), CVS.*

---

### SCM
**Source Code Management** — управление исходным кодом.  
Синоним [VCS](#vcs), используется преимущественно в корпоративной среде.

---

### SHA
**Secure Hash Algorithm** — алгоритм безопасного хэширования.  
Git использует SHA-1 (40 символов) или SHA-256 (64 символа) для уникальной идентификации каждого коммита, дерева и объекта.  
Пример SHA-1: `a3f5c2d8e1b4...` (40 hex-символов).

---

### HEAD
**HEAD** — специальный указатель в Git, обозначающий «где я нахожусь сейчас».  
Обычно указывает на последний коммит текущей ветки.  
При состоянии **detached HEAD** указывает напрямую на коммит, а не на ветку.

---

### PR
**Pull Request** — запрос на принятие изменений (термин GitHub/Bitbucket).  
Разработчик просит ревьювера проверить его ветку и слить её в целевую.  
→ *На GitLab называется [MR](#mr).*

---

### MR
**Merge Request** — запрос на слияние (термин GitLab).  
Эквивалент [PR](#pr) на платформе GitLab.

---

### CICD
**CI/CD** — Continuous Integration / Continuous Delivery (или Deployment).

- **CI (Continuous Integration)** — автоматическая сборка и тестирование каждого коммита/PR.
- **CD (Continuous Delivery)** — автоматическая подготовка к выкатке на прод (ручное подтверждение).
- **CD (Continuous Deployment)** — полностью автоматическая выкатка на прод без ручного шага.

---

### WIP
**Work In Progress** — «в работе», незавершённый PR/MR.  
Обычно ставится в заголовок PR: `WIP: Add login form` — сигнализирует, что код ещё не готов к ревью.  
GitHub поддерживает Draft PR как официальный эквивалент.

---

### LGTM
**Looks Good To Me** — «выглядит хорошо для меня».  
Стандартный ответ ревьювера в комментариях к PR, означающий одобрение изменений.

---

### ACK
**Acknowledged** или **Accepted** — подтверждение/принятие.  
В ревью кода означает согласие с изменениями.  
→ *Противоположность: [NACK](#nack).*

---

### NACK
**Not Acknowledged** — отклонение изменений.  
Ревьювер возражает против принятия изменений в текущем виде.

---

### RFC
**Request For Comments** — запрос на обратную связь.  
В контексте Git/GitHub: черновой PR с пометкой `RFC:` приглашает других участников обсудить подход до начала финальной реализации.

---

### GPG
**GNU Privacy Guard** — инструмент шифрования и подписи.  
Используется для **подписи коммитов и тегов**, чтобы подтвердить авторство:  
```bash
git commit -S -m "Подписанный коммит"
```

---

### SSH
**Secure Shell** — протокол безопасного удалённого соединения.  
Git использует SSH для аутентификации с удалёнными репозиториями вместо пароля:  
```
git@github.com:user/repo.git
```

---

### HTTPS
**HyperText Transfer Protocol Secure** — защищённый HTTP.  
Альтернативный способ подключения к удалённым репозиториям (через логин/пароль или токен):  
```
https://github.com/user/repo.git
```

---

### URL
**Uniform Resource Locator** — унифицированный указатель ресурса (ссылка).  
В Git: адрес удалённого репозитория.

---

### API
**Application Programming Interface** — программный интерфейс приложения.  
GitHub/GitLab предоставляют REST API для автоматизации: создания PR, управления issues, запуска CI и т.д.

---

### CLI
**Command Line Interface** — интерфейс командной строки.  
Git по умолчанию управляется через [CLI](#cli): `git commit`, `git push` и т.д.

---

### GUI
**Graphical User Interface** — графический пользовательский интерфейс.  
Программы-клиенты Git с визуальным интерфейсом: GitKraken, SourceTree, GitHub Desktop, встроенный инструмент в VS Code.

---

### IDE
**Integrated Development Environment** — интегрированная среда разработки.  
Большинство современных IDE (VS Code, CLion, IntelliJ) имеют встроенную поддержку Git.

---

## Термины Git (алфавитный порядок)

| Термин | Определение |
|---|---|
| **Bare repository** | Репозиторий без рабочей директории — только объекты Git. Используется как центральный сервер. |
| **Blob** | Внутренний объект Git, хранящий содержимое файла. |
| **Branch** (ветка) | Лёгкий указатель на конкретный коммит. Позволяет вести параллельную разработку. |
| **Checkout** | Переключение на ветку или коммит; обновление рабочей директории. |
| **Cherry-pick** | Применение одного конкретного коммита из другой ветки. |
| **Clone** | Создание полной локальной копии удалённого репозитория. |
| **Commit** | Снимок (snapshot) состояния файлов в конкретный момент времени. Неизменяем. |
| **Conflict** (конфликт) | Ситуация, когда Git не может автоматически слить изменения двух веток в одном месте файла. |
| **Fast-forward** | Слияние, при котором целевая ветка просто «перематывается» вперёд без создания коммита слияния. |
| **Fetch** | Загрузка изменений из удалённого репозитория без применения к рабочей директории. |
| **Fork** | Копия чужого репозитория в вашем аккаунте на хостинге (GitHub/GitLab). |
| **HEAD** | Указатель на текущее местоположение в истории. |
| **Index** | Другое название для Staging Area (индекс). |
| **Merge** | Слияние двух веток в одну. |
| **Object** | Базовая единица хранения в Git: blob, tree, commit, tag. |
| **Origin** | Стандартное имя по умолчанию для основного удалённого репозитория. |
| **Pull** | `fetch` + `merge` (или `rebase`): получить и применить изменения из удалённого репозитория. |
| **Push** | Отправка локальных коммитов в удалённый репозиторий. |
| **Rebase** | Перенос серии коммитов на новое основание; переписывает историю. |
| **Reflog** | Журнал всех изменений [HEAD](#head) — «машина времени» Git. |
| **Remote** | Удалённый репозиторий (сервер). |
| **Repository** (репозиторий) | Папка `.git` + рабочая директория; база данных всей истории проекта. |
| **Reset** | Перемещение HEAD и/или ветки на другой коммит. |
| **Revert** | Создание нового коммита, отменяющего изменения предыдущего. |
| **Staging Area** | Промежуточная зона между рабочей директорией и коммитом. |
| **Stash** | Временное «складирование» незафиксированных изменений. |
| **Tag** | Постоянная именованная метка на конкретном коммите (обычно — версия релиза). |
| **Tree** | Внутренний объект Git, представляющий директорию. |
| **Upstream** | 1) Ветка, на которую настроено отслеживание. 2) Оригинальный репозиторий, от которого был сделан fork. |
| **Working Directory** | Файлы проекта, которые вы видите и редактируете в файловой системе. |
