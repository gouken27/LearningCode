## 1. Что такое система контроля версий

### Проблема без VCS

Представьте: вы пишете программу одни. Вы меняете `main.cpp`, что-то ломается, и вы не помните, что именно изменили час назад. Теперь представьте команду из десяти человек, где каждый редактирует одни и те же файлы.

Без инструмента контроля версий возникают типичные проблемы:

```
project/
├── main_FINAL.cpp
├── main_FINAL_v2.cpp
├── main_FINAL_v2_REALLY.cpp
├── main_backup_27_april.cpp
└── main_working_DO_NOT_DELETE.cpp
```

Это — ручная «система контроля версий». Она ломается при любом масштабе.

### Что решает VCS

[VCS](./glossary.md#vcs) — программа, которая:

- **Запоминает историю**: каждое изменение файлов фиксируется как коммит со временем и автором
- **Позволяет откатываться**: вернуться к любому моменту прошлого в один клик
- **Разрешает конфликты**: когда два человека изменили одно место в файле, VCS помогает их слить
- **Позволяет работать параллельно**: разные функции разрабатываются в отдельных ветках, не мешая друг другу
- **Сохраняет «почему»**: к каждому изменению прикладывается сообщение с объяснением

### Три поколения VCS

```
┌──────────────────────────────────────────────────────────────────┐
│                    ЭВОЛЮЦИЯ VCS                                  │
├─────────────────┬──────────────────┬────────────────────────────┤
│  1-е поколение  │  2-е поколение   │  3-е поколение             │
│  Локальные VCS  │  CVCS            │  DVCS                      │
│  (RCS, SCCS)    │  (SVN, CVS)      │  (Git, Mercurial)          │
├─────────────────┼──────────────────┼────────────────────────────┤
│  История только │  Центральный     │  Полная история у          │
│  на вашем ПК    │  сервер хранит   │  каждого участника         │
│                 │  всю историю     │                            │
├─────────────────┼──────────────────┼────────────────────────────┤
│  ✓ Просто       │  ✓ Совместная    │  ✓ Работа офлайн           │
│  ✗ Нет          │    работа        │  ✓ Нет единой точки        │
│    совместной   │  ✗ Сервер упал → │    отказа                  │
│    работы       │    работа стоит  │  ✓ Скорость                │
└─────────────────┴──────────────────┴────────────────────────────┘
```

### Централизованная vs Распределённая

```
       CVCS (SVN)                        DVCS (Git)
  ┌─────────────────┐              ┌─────────────────┐
  │  Центральный    │              │  Удалённый      │
  │  сервер         │              │  репозиторий    │
  │  (вся история)  │              │  (GitHub и др.) │
  └────────┬────────┘              └────────┬────────┘
           │ checkout/commit                │ push/pull
    ┌──────┼──────┐               ┌─────────┼─────────┐
    ↓      ↓      ↓               ↓         ↓         ↓
  [Аня] [Боря] [Вася]         [Аня]     [Боря]    [Вася]
  только  только  только      полная    полная    полная
  рабочая рабочая рабочая     история   история   история
  папка   папка   папка
```

**Ключевое отличие**: в [DVCS](./glossary.md#dvcs) у каждого разработчика — **полный клон** репозитория. Это означает: работа в самолёте, быстрые локальные операции, резервная копия у каждого участника.

### Три зоны Git

Понимание трёх зон — основа работы с Git. **Почему вообще три зоны, а не просто «изменил → сохранил»?** Потому что Git даёт вам контроль: вы можете изменить десять файлов, но в один коммит положить только три — те, которые относятся к одной задаче. Staging Area — это «стол сборки» коммита.

```
┌──────────────────────────────────────────────────────────────────────────┐
│                          ТРИ ЗОНЫ GIT                                    │
│                                                                          │
│  ЗОНА 1                   ЗОНА 2                  ЗОНА 3                 │
│  Working Directory        Staging Area            Repository             │
│  (Рабочая директория)     (Индекс / Область       (.git/)                │
│                            подготовки)                                   │
│  ┌─────────────────┐      ┌─────────────────┐    ┌─────────────────┐    │
│  │  Файлы на диске │      │  «Черновик»      │    │  История всех   │    │
│  │  — то, что вы   │      │  следующего      │    │  коммитов.      │    │
│  │  видите в папке │      │  коммита.        │    │  Неизменяема.   │    │
│  │  проекта и      │      │  Можно добавлять │    │  Хранится в     │    │
│  │  редактируете   │      │  и убирать файлы │    │  скрытой папке  │    │
│  │  в редакторе.   │      │  сколько угодно  │    │  .git/          │    │
│  │                 │      │  до коммита.     │    │                 │    │
│  └────────┬────────┘      └────────┬─────────┘    └─────────────────┘    │
│           │                        │                        ▲             │
│           │  git add <file>        │                        │             │
│           │  ────────────────────► │    git commit -m "..." │             │
│           │                        │  ──────────────────────┘             │
│           │  git restore --staged  │                                      │
│           │  ◄──────────────────── │   (убрать из staging,                │
│           │                        │    изменения остаются в файле)       │
│           │  git restore <file>                                            │
│           │  ◄─────────────────────────────────────────────               │
│           │  (отменить изменения в файле, вернув к последнему коммиту)    │
└──────────────────────────────────────────────────────────────────────────┘
```

**Что делает каждая команда:**

- `git add <file>` — «я хочу включить этот файл в следующий коммит»; перемещает снимок файла из рабочей директории в Staging Area
- `git restore --staged <file>` — «передумал, убери из черновика»; файл на диске не меняется, только убирается из Staging Area
- `git restore <file>` — «выброси мои изменения, верни файл как в последнем коммите»; изменения в рабочей директории безвозвратно теряются
- `git commit -m "..."` — «зафиксируй черновик»; всё из Staging Area становится новым коммитом в Repository

## 2. Онлайн-сервисы: удалённые репозитории

### Что такое удалённый репозиторий

Удалённый репозиторий ([Remote](./glossary.md#термины-git-алфавитный-порядок)) — это копия вашего репозитория, расположенная на сервере в интернете. Он выполняет три задачи:

1. **Резервная копия** — ваш ноутбук сгорел, код жив на сервере
2. **Точка синхронизации** — команда обменивается изменениями через него
3. **Витрина** — другие могут видеть и использовать ваш код

### Основные платформы

```
┌─────────────────────────────────────────────────────────────────┐
│                  ОНЛАЙН-ХОСТИНГИ GIT                            │
├──────────────┬──────────────┬──────────────┬────────────────────┤
│   GitHub     │   GitLab     │  Bitbucket   │  Gitea / Forgejo   │
├──────────────┼──────────────┼──────────────┼────────────────────┤
│ github.com   │ gitlab.com   │ bitbucket.   │ Self-hosted        │
│              │              │ org          │ (свой сервер)      │
├──────────────┼──────────────┼──────────────┼────────────────────┤
│ • Крупнейший │ • Встроенный │ • Хорошо     │ • Полный контроль  │
│   хостинг    │   CI/CD      │   интегрир.  │ • Без зависимости  │
│ • GitHub     │ • Self-hosted│   с Jira     │   от внешних       │
│   Actions    │   возможен   │ • Atlassian  │   сервисов         │
│ • Copilot    │ • MR вместо  │   экосистема │                    │
│              │   PR         │              │                    │
├──────────────┼──────────────┼──────────────┼────────────────────┤
│ Бесплатно    │ Бесплатно    │ Бесплатно    │ Бесплатно          │
│ для публ. и  │ для публ.    │ для малых    │ (надо свой         │
│ части приват.│ репозиториев │ команд       │ сервер)            │
└──────────────┴──────────────┴──────────────┴────────────────────┘
```

### Как связать локальный репозиторий с удалённым

```bash
# Посмотреть текущие удалённые репозитории
git remote -v

# Добавить удалённый репозиторий (origin — стандартное имя)
git remote add origin https://github.com/user/repo.git

# Удалить
git remote remove origin

# Переименовать
git remote rename origin upstream
```

### SSH vs HTTPS — два способа подключения к удалённому репозиторию

Когда вы делаете `git clone`, `git push` или `git pull`, Git должен как-то **аутентифицироваться** на сервере — доказать, что вы — это вы. Для этого существует два протокола: HTTPS и SSH.

```
ЧТО ТАКОЕ HTTPS В GIT:
  URL:  https://github.com/user/repo.git
  Это тот же протокол, что и сайты в браузере.
  Аутентификация: логин + пароль, или (современный стандарт)
                  личный токен доступа (Personal Access Token, PAT).

ЧТО ТАКОЕ SSH В GIT:
  URL:  git@github.com:user/repo.git
  Это криптографический протокол безопасного соединения.
  Аутентификация: пара ключей (публичный + приватный).
```

**Что такое SSH-ключ?**  
SSH-ключ — это пара файлов, которую генерирует ваш компьютер:

```
Приватный ключ (~/.ssh/id_ed25519):    Публичный ключ (~/.ssh/id_ed25519.pub):
──────────────────────────────────     ──────────────────────────────────────
-----BEGIN OPENSSH PRIVATE KEY-----    ssh-ed25519 AAAAC3Nza... your@email.com
b3BlbnNzaC1rZXktdjEAAAA...
-----END OPENSSH PRIVATE KEY-----

ТОЛЬКО У ВАС.                          Загружается на GitHub один раз.
Никому не показывать.                  Это как замок на двери.
Это как ключ от замка.
```

Принцип работы: GitHub хранит ваш публичный ключ (замок). При подключении ваш компьютер доказывает, что у него есть приватный ключ (не показывая его) — через математическое доказательство. Пароль по сети не передаётся никогда.

```
┌──────────────────────────────────────────────────────────────────┐
│                    HTTPS vs SSH                                   │
├───────────────────────────────┬──────────────────────────────────┤
│         HTTPS                 │         SSH                      │
├───────────────────────────────┼──────────────────────────────────┤
│ Адрес: https://github.com/... │ Адрес: git@github.com:...        │
├───────────────────────────────┼──────────────────────────────────┤
│ ✓ Работает сразу, без         │ ✓ Один раз настроил — больше     │
│   настройки (за фаерволами,   │   никаких паролей и токенов      │
│   в корпоративных сетях,      │ ✓ Безопаснее: приватный ключ     │
│   в общежитиях)               │   никогда не покидает ваш ПК     │
│ ✓ Не нужно настраивать ключи  │ ✓ Стандарт для профессионалов    │
│                               │   и CI/CD систем                 │
│ ✗ Нужен токен (PAT) при       │ ✗ Требует начальной настройки    │
│   каждой операции (без        │ ✗ Порт 22 бывает заблокирован    │
│   менеджера паролей)          │   в некоторых сетях              │
│ ✗ Токен можно случайно        │                                  │
│   засветить в логах           │                                  │
├───────────────────────────────┼──────────────────────────────────┤
│ Когда использовать:           │ Когда использовать:              │
│ • Вы за корпоративным         │ • Это ваша личная машина         │
│   прокси или фаерволом        │ • Вы часто пушите код            │
│ • Разовое клонирование        │ • Настраиваете CI/CD             │
│   чужого публичного репо      │ • Хотите не вводить пароль       │
└───────────────────────────────┴──────────────────────────────────┘
```

**Пример 1 — HTTPS, разовая операция:**  
Вы хотите скачать открытый проект, посмотреть код, возможно предложить правку. Вы не планируете работать с ним постоянно. HTTPS — быстрее: `git clone https://github.com/torvalds/linux.git` — никаких ключей, никакой настройки.

**Пример 2 — SSH, ежедневная работа:**  
Вы работаете в компании и каждый день делаете 10+ push'ей в рабочий репозиторий. С SSH вы один раз добавляете ключ на GitHub — и больше никогда не вводите пароль. С HTTPS каждый `git push` требует токен (или настройки keychain).

```bash
# Генерация SSH-ключа (один раз на компьютер):
ssh-keygen -t ed25519 -C "your@email.com"
# -t ed25519  — алгоритм (современный, быстрый, безопасный)
# -C          — комментарий (обычно ваш email), чтобы потом понять чей ключ

# Посмотреть публичный ключ (его нужно добавить на GitHub):
cat ~/.ssh/id_ed25519.pub

# Проверить подключение к GitHub:
ssh -T git@github.com
# Ответ: Hi username! You've successfully authenticated...

# На GitHub: Settings → SSH and GPG keys → New SSH key → вставьте содержимое .pub файла
```

## 3. Установка Git

### Windows

Скачайте установщик с https://git-scm.com/download/win

**Что даёт установщик Git for Windows:**

```
┌─────────────────────────────────────────────────────────────┐
│              ЧТО УСТАНАВЛИВАЕТ GIT FOR WINDOWS              │
├─────────────────────────────────────────────────────────────┤
│  ✓ git.exe          — сам Git                               │
│  ✓ Git Bash         — терминал с bash (Unix-команды         │
│                        на Windows: ls, cat, grep, ...)      │
│  ✓ Git CMD          — git в стандартной командной строке    │
│  ✓ Git GUI          — простейший графический интерфейс      │
│  ✓ Git Credential   — менеджер паролей/токенов              │
│    Manager (GCM)      (не нужно вводить токен каждый раз)   │
│  ✓ OpenSSH          — генерация SSH-ключей                  │
│  ✓ MinTTY           — терминальный эмулятор для Git Bash    │
└─────────────────────────────────────────────────────────────┘
```

**Важные опции установщика:**

```
  "Adjusting your PATH environment"
  → Выберите: "Git from the command line and also from 3rd-party software"
    (git будет доступен в PowerShell и CMD, не только в Git Bash)

  "Choosing the default editor"
  → VS Code, если установлен. Или Notepad++.
    НЕ выбирайте Vim, если не знакомы с ним.

  "Choosing the default branch name"
  → Выберите "main" (современный стандарт вместо "master")

  "Line ending conversions" — что это и зачем:
  ┌─────────────────────────────────────────────────────────────────────┐
  │  Проблема: Windows и Linux/macOS используют разные символы конца    │
  │  строки в текстовых файлах.                                         │
  │                                                                     │
  │  Windows:    \r\n  (два символа: CR + LF, «возврат каретки + перевод│
  │                     строки») — наследие печатных машинок             │
  │  Linux/macOS: \n   (один символ: LF, «перевод строки»)              │
  │                                                                     │
  │  Если в команде есть разработчики на Windows и Linux, и каждый      │
  │  сохраняет файлы в своём формате — git diff будет показывать        │
  │  «изменения» в каждой строке файла, хотя реальных правок не было.   │
  │  Это засоряет историю и ломает code review.                         │
  │                                                                     │
  │  Решение: хранить в репозитории всегда \n (Unix-стиль),             │
  │  а Git автоматически конвертирует при checkout/commit.              │
  └─────────────────────────────────────────────────────────────────────┘

  → Windows: "Checkout Windows-style, commit Unix-style"
    При git checkout: \n → \r\n (для вашего редактора)
    При git commit:   \r\n → \n  (в репозиторий идёт \n)
    Результат: в репозитории всегда \n, на вашем ПК всегда \r\n.

  → Linux/macOS: "Checkout as-is, commit Unix-style"
    Файлы уже в \n — конвертировать при checkout не нужно.
    При commit просто убедится, что нет случайных \r\n.
```

### Первоначальная настройка (обязательно после установки)

```bash
# ─── ИМЯ И EMAIL ─────────────────────────────────────────────────────────
# Зачем: Git встраивает эти данные в КАЖДЫЙ коммит навсегда.
# Имя и email видны всем: в git log, на GitHub, в code review.
# По email GitHub связывает коммит с вашим аккаунтом и показывает аватар.
# Если email не совпадает с аккаунтом GitHub — коммиты будут «безымянными».
git config --global user.name "Иван Иванов"
git config --global user.email "ivan@example.com"

# ─── РЕДАКТОР ────────────────────────────────────────────────────────────
# Зачем: Git открывает редактор при git commit (без -m), git rebase -i,
# разрешении конфликтов и других операциях. Без настройки откроется vim,
# и неопытный пользователь не знает, как из него выйти (Esc, затем :q!).
git config --global core.editor "code --wait"    # VS Code
# git config --global core.editor "nano"         # для тех, кто на сервере

# ─── ВЕТКА ПО УМОЛЧАНИЮ ──────────────────────────────────────────────────
# Зачем: исторически Git называл главную ветку "master".
# Современный стандарт — "main". Если не настроить, старые версии Git
# создадут ветку "master", и GitHub предупредит о несоответствии.
git config --global init.defaultBranch main

# ─── КРАСИВЫЙ ЛОГ ────────────────────────────────────────────────────────
# Зачем: стандартный git log показывает много лишнего. Этот формат
# выводит компактную однострочную историю с графом веток.
git config --global log.oneline true

# ─── ГДЕ ХРАНИТСЯ КОНФИГУРАЦИЯ ───────────────────────────────────────────
# ~/.gitconfig        — глобальная (--global): для всех ваших репозиториев
# .git/config         — локальная (--local):  только для текущего репо
# /etc/gitconfig      — системная (--system): для всех пользователей ОС
# При конфликте: локальная побеждает глобальную, глобальная — системную.

# ─── ПРОВЕРИТЬ ВСЁ ───────────────────────────────────────────────────────
git config --list                    # все настройки (все уровни)
git config --list --show-origin      # + откуда берётся каждая настройка
git config user.email                # посмотреть конкретный параметр
```

## 4. Первые шаги: инициализация и первый коммит

### Анатомия команды Git

Прежде чем идти дальше — разберём, из чего состоит любая команда Git:

```
  git        commit      -m        "Initial commit"
  ───────    ──────────  ──────    ─────────────────
  Программа  Подкоманда  Ключ/-    Аргумент ключа
  (пакет     (что        флаг      (значение для -m:
  git)       делаем)     (опция)    текст сообщения)

  git        log         --oneline
  ───────    ──────────  ──────────────
  Программа  Подкоманда  Длинный ключ  (двойное тире)
                         = краткий вывод, без аргумента

  git        remote      -v
  ───────    ──────────  ──
  Программа  Подкоманда  Краткий ключ (одно тире + буква)
                         -v = --verbose = подробный вывод
```

Правило простое:
- `git` — это программа, к которой обращаетесь
- Следующее слово — **подкоманда**: `init`, `add`, `commit`, `log`, `branch`...
- Ключи с `-` (одна черта) — краткие: `-m`, `-v`, `-a`, `-u`
- Ключи с `--` (две черты) — длинные: `--oneline`, `--staged`, `--hard`
- После ключа может идти аргумент: `-m "текст"`, `--author="Иван"`

### Создание нового репозитория

```bash
mkdir my-project          # создать папку (mkdir = make directory, аргумент — имя папки)
cd my-project             # войти в папку (cd = change directory)

git init                  # инициализировать репозиторий
# git    — программа
# init   — подкоманда «инициализировать»
# Что происходит: создаётся скрытая папка .git/ внутри текущей директории.
# Эта папка — и есть весь Git: история коммитов, настройки, объекты.
# Удалите .git/ — репозиторий исчезнет, файлы останутся.

git status                # посмотреть состояние рабочей директории
# git    — программа
# status — подкоманда «показать статус»
# Показывает: какие файлы изменены, что в staging, на какой ветке вы
```

### Жизненный цикл файла в Git

В системе контроля версий каждый файл имеет своё **состояние** — Git постоянно отслеживает, что происходит с каждым файлом. Это необходимо, чтобы знать: что изменилось, что нужно включить в следующий коммит, а что вообще не трогать.

Существуют четыре состояния:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     СОСТОЯНИЯ ФАЙЛА В GIT                                   │
│                                                                             │
│  Untracked          Unmodified          Modified            Staged          │
│  ──────────         ──────────          ──────────          ──────────      │
│  Git не знает       Git знает о нём,    Файл изменён        Файл готов      │
│  об этом            изменений нет       с момента           к коммиту       │
│  файле совсем       (совпадает с        последнего          (добавлен       │
│  (новый файл,       последним           коммита             в Staging       │
│  ещё ни разу        коммитом)                               Area)           │
│  не добавлялся)                                                             │
│       │                  │                   │                  │           │
│       │  git add <file>  │                   │   git add <file> │           │
│       └──────────────────┼───────────────────┼────────────────► │           │
│                          │                   │                  │           │
│       ◄──────────────────┼───────────────────┼──────────────────┘           │
│  git rm --cached         │  (редактируем      │   git restore --staged       │
│  (убрать из              │   файл в           │   (убрать из staging,        │
│   отслеживания)          │   редакторе)       │    файл остаётся изменённым) │
│                          │                   │                              │
│                          └───────────────────►│                             │
│                                               │  git restore <file>         │
│                          ◄────────────────────┘  (отменить изменения,       │
│                          (после git commit:       вернуть к Unmodified)     │
│                           файл снова Unmodified)                            │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Что делает каждая команда на схеме:**

| Команда | Откуда → Куда | Что происходит |
|---|---|---|
| `git add <file>` | Untracked или Modified → Staged | Снимок файла кладётся в Staging Area |
| `git commit` | Staged → Unmodified | Создаётся коммит, файл теперь «чистый» |
| `git restore --staged <file>` | Staged → Modified | Убирает из staging, файл на диске не меняется |
| `git restore <file>` | Modified → Unmodified | Выбрасывает изменения, возвращает файл к последнему коммиту |
| `git rm --cached <file>` | Unmodified → Untracked | Перестаёт отслеживать файл (сам файл остаётся) |

### Первый коммит — пошагово

```bash
# 1. Создать файл
echo "# Мой проект" > README.md

# 2. Посмотреть статус — файл Untracked
git status

# 3. Добавить в Staging Area
git add README.md
# Или добавить всё:
git add .

# 4. Посмотреть, что готово к коммиту
git status
git diff --staged   # показывает diff для staged файлов

# 5. Создать коммит
git commit -m "Initial commit: добавлен README"

# 6. Посмотреть историю
git log
git log --oneline   # компактный вид
```

### Хороший commit message

Сообщение коммита — это объяснение для вашего будущего «я» и для коллег. Через полгода вы откроете `git log` и должны за 3 секунды понять, что делал каждый коммит.

**Стандарт Conventional Commits** — популярное соглашение о формате:

```
  тип(область): краткое описание что сделано
  ───────────── ──────────────────────────────
       │              │
       │              └── Пишется с маленькой буквы, без точки в конце.
       │                  До 72 символов — иначе обрезается в git log.
       │                  Описывает ЧТО изменено, а не КАК.
       │
       └── Тип — обязательная метка из фиксированного набора:

  feat     — новая функциональность для пользователя
  fix      — исправление бага
  docs     — изменения только в документации
  refactor — рефакторинг (не баг, не фича, просто улучшение кода)
  test     — добавление или изменение тестов
  chore    — служебные задачи (обновление зависимостей, настройка CI)
  style    — форматирование кода (пробелы, отступы), без логических изменений
  perf     — улучшение производительности

  Область (необязательна) — подсистема, которую затронули:
  feat(greeter): ...      fix(math_utils): ...      docs(readme): ...
```

**Примеры:**

```
feat: добавлен модуль математических утилит
fix(greeter): исправлен краш при пустом имени
docs: добавлен раздел об установке в README
refactor(math_utils): factorial переписан итеративно
chore: обновлён CMakeLists до минимальной версии 3.20
```

**Зачем тип в начале?** Потому что `git log --oneline` покажет сотни строк — по префиксу сразу видно природу изменения. Инструменты вроде `conventional-changelog` умеют автоматически генерировать CHANGELOG из таких коммитов.

```
ХОРОШО:                              ПЛОХО:
feat: добавлен метод sayGoodbye      fix
fix: исправлено деление на ноль      правки
docs: описан git rebase              аааааа изменил файлы
refactor: упрощена логика ветвления  работает
```

## 6. Ветки и слияния

### Что такое ветка и HEAD

**HEAD** — специальный указатель, который всегда отвечает на вопрос «где я нахожусь сейчас». Обычно HEAD указывает на **ветку**, а ветка указывает на **коммит**:

```
  HEAD → main → коммит C
                 ↑
          «ты здесь»
```

**Ветка** — это не папка, не копия файлов. Это просто **файл с одним SHA-хэшем** внутри `.git/refs/heads/`. Создать ветку = записать хэш одного коммита в текстовый файл. Поэтому это занимает миллисекунды.

```
.git/refs/heads/main          .git/refs/heads/feature
───────────────────────       ─────────────────────────────
a3f5c2d8e1b4...               d7e9f1a2b3c4...
(SHA последнего               (SHA последнего коммита
 коммита в main)               в feature)
```

**Detached HEAD** — когда HEAD указывает напрямую на коммит, минуя ветку. Это происходит при `git checkout <SHA>`. В этом состоянии можно делать коммиты, но при переключении ветки они станут «висячими» и могут быть удалены GC. Выход: `git switch -c new-branch` — создать ветку прямо из detached HEAD.

```
  main:    A ── B ── C
                      ↑
                     HEAD

После git checkout -b feature:
  main:    A ── B ── C
                      \
  feature:             C  ← HEAD теперь здесь
```

### Основные команды веток

```bash
# Посмотреть все ветки (* = текущая)
git branch

# Создать новую ветку
git branch feature/add-farewell

# Переключиться на ветку
git checkout feature/add-farewell
# Современный способ (Git 2.23+):
git switch feature/add-farewell

# Создать и переключиться одновременно
git checkout -b feature/add-farewell
# или:
git switch -c feature/add-farewell

# Удалить ветку (только если слита)
git branch -d feature/add-farewell
# Принудительно удалить
git branch -D feature/add-farewell

# Переименовать текущую ветку
git branch -m new-name
```

### Пример рабочего процесса с ветками

**Добавим метод `sayGoodbye()` к нашему C++ проекту в отдельной ветке:**

```bash
# 1. Создать ветку для новой функции
git switch -c feature/add-farewell

# 2. Изменить greeter.h — добавить объявление
# void sayGoodbye() const;

# 3. Изменить greeter.cpp — добавить реализацию
# void Greeter::sayGoodbye() const {
#     std::cout << "До свидания, " << m_name << "!" << std::endl;
# }

# 4. Зафиксировать изменения
git add greeter.h greeter.cpp
git commit -m "feat(greeter): добавлен метод sayGoodbye()"

# 5. Переключиться обратно на main
git switch main

# 6. Слить ветку feature в main
git merge feature/add-farewell

# 7. Удалить ветку — она выполнила свою задачу
git branch -d feature/add-farewell
```

### Визуализация веток: типичная история проекта

```
      A ─── B ─── C ─────────── G  ← main
                   \           /
                    D ─── E ─ F  ← feature/add-farewell
                    
Где:
  A = "Initial commit"
  B = "feat: добавлен README"
  C = "feat: начальная структура C++ проекта"
  D = "feat(greeter): объявлен метод sayGoodbye"
  E = "feat(greeter): реализован sayGoodbye"
  F = "test: добавлен тест sayGoodbye"
  G = merge commit "Merge feature/add-farewell into main"
```

### Типы слияния: Fast-forward vs Merge commit

```
FAST-FORWARD (когда main не ушёл вперёд):
  До:   main → C, feature → E
              A ─ B ─ C
                       \
                        D ─ E ← feature

  После git merge feature:
              A ─ B ─ C ─ D ─ E ← main (просто перемотан)
              
  (нет отдельного merge-коммита, история линейная)

──────────────────────────────────────────────────

3-WAY MERGE (когда main ушёл вперёд):
  До:   
              A ─ B ─ C ─ F ← main
                       \
                        D ─ E ← feature

  После git merge feature:
              A ─ B ─ C ─ F ─ G ← main
                       \       /
                        D ─ E

  G = merge-коммит (имеет двух родителей: F и E)
```

### Конфликты при слиянии

Конфликт возникает, когда две ветки изменили **одно и то же место** в файле:

```bash
# Git сообщит:
# CONFLICT (content): Merge conflict in greeter.cpp
# Automatic merge failed; fix conflicts and then commit.

# Содержимое файла при конфликте:
<<<<<<< HEAD          ← ваши изменения (main)
    std::cout << "Привет, " << m_name << "!\n";
=======               ← разделитель
    std::cout << "Hello, " << m_name << "!\n";
>>>>>>> feature/english-greeting  ← изменения из ветки

# Решение:
# 1. Откройте файл и выберите нужный вариант (или объедините оба)
# 2. Удалите маркеры <<<<<, =====, >>>>>
# 3. Зафиксируйте результат:
git add greeter.cpp
git commit -m "merge: объединены ветки приветствий"
```

### Rebase: альтернатива merge

```
MERGE:                         REBASE:
A─B─C─F─G (main)              A─B─C─F─D'─E' (main/feature)
       \ /                    (история линейная!)
        ↑ merge-commit
        D─E (feature)

git rebase main:
  1. Git находит общего предка (C)
  2. Снимает коммиты D и E
  3. «Воспроизводит» их поверх F как D' и E'
  4. D' и E' — новые коммиты с новыми SHA!
```

```bash
# Находясь на feature:
git rebase main

# Интерактивный rebase (редактировать историю):
git rebase -i HEAD~3   # последние 3 коммита
```

> ⚠️ **Правило**: никогда не делайте rebase для коммитов, которые уже были запушены в публичную ветку. Rebase переписывает историю.

## 7. Работа с коммитами: stash, revert, reset, reflog

### Git Stash — временное «складирование»

Stash — «рюкзак» для незафиксированных изменений. Полезен, когда нужно срочно переключиться на другую ветку, а текущая работа не готова к коммиту.

```bash
# Сохранить изменения в stash
git stash
# С описанием (рекомендуется!):
git stash push -m "WIP: реализация sayGoodbye, не закончено"

# Посмотреть список stash
git stash list
# stash@{0}: On feature: WIP: реализация sayGoodbye, не закончено
# stash@{1}: On main: черновик README

# Применить последний stash (сохраняя его в списке)
git stash apply

# Применить конкретный stash
git stash apply stash@{1}

# Применить и удалить из списка
git stash pop

# Применить конкретный и удалить
git stash pop stash@{0}

# Удалить конкретный stash
git stash drop stash@{0}

# Удалить все stash
git stash clear

# Создать ветку из stash (мощная функция!)
git stash branch feature/from-stash stash@{0}

# Посмотреть изменения в stash без применения
git stash show -p stash@{0}
```

### Git Stash List — полный обзор

```
┌─────────────────────────────────────────────────────────────┐
│                    УПРАВЛЕНИЕ STASH                         │
│                                                             │
│  git stash list                                             │
│  ─────────────────────────────────────────────             │
│  stash@{0}: On main: WIP feat X           ← самый свежий   │
│  stash@{1}: On feature/y: черновик        ← старше         │
│  stash@{2}: WIP: незаконченный рефактор   ← самый старый   │
│                                                             │
│  Stash — это стек (LIFO):                                   │
│  • push помещает наверх (становится stash@{0})              │
│  • pop берёт с верха (stash@{0})                            │
│  • Все остальные сдвигаются: {0}→{1}, {1}→{2} при push     │
└─────────────────────────────────────────────────────────────┘
```

### Git Reset — перемотка истории

Reset — перемещение указателя ветки (и, возможно, HEAD) на другой коммит.

```
История: A ─── B ─── C ─── D ← HEAD (main)

git reset HEAD~1  (отмотать на 1 коммит назад):

git reset --soft HEAD~1:
  A ─── B ─── C ← HEAD
  Изменения из D → в Staging Area (готовы к коммиту)
  ✓ Изменения сохранены, просто «расфиксированы»

git reset --mixed HEAD~1 (по умолчанию):
  A ─── B ─── C ← HEAD
  Изменения из D → в Working Directory (не staged)
  ✓ Изменения сохранены в файлах

git reset --hard HEAD~1:
  A ─── B ─── C ← HEAD
  Изменения из D → УДАЛЕНЫ
  ✗ Данные потеряны (осторожно!)
```

```bash
# Примеры использования reset:

# Отменить последний коммит, оставив изменения (staged)
git reset --soft HEAD~1

# Отменить последний коммит, оставив изменения (unstaged)
git reset HEAD~1

# Убрать файл из Staging Area (не меняя файл на диске)
git reset HEAD greeter.cpp
# Современный синтаксис:
git restore --staged greeter.cpp

# Сбросить к конкретному коммиту
git reset --hard abc1234

# ВАЖНО: после --hard потерянные коммиты можно найти через reflog
```

### Git Revert — безопасная отмена

Revert создаёт **новый коммит**, отменяющий изменения старого. История не переписывается — это безопасно для публичных веток.

```
История:  A ─── B ─── C ─── D ← HEAD

git revert C:
  A ─── B ─── C ─── D ─── C' ← HEAD
                           ↑
                    новый коммит, который отменяет C

git revert D..HEAD   (отменить несколько)
git revert --no-commit C D   (подготовить отмену без коммита)
```

```bash
# Отменить конкретный коммит
git revert abc1234

# Отменить последний коммит
git revert HEAD

# Отменить без автоматического открытия редактора
git revert --no-edit HEAD

# Отменить серию коммитов (не создавая коммит сразу)
git revert --no-commit HEAD~3..HEAD
git commit -m "revert: отмена последних 3 коммитов"
```

### Reset vs Revert: когда что использовать

```
┌────────────────────────────────────────────────────────────────┐
│              RESET vs REVERT                                    │
├───────────────────────┬────────────────────────────────────────┤
│  git reset            │  git revert                            │
├───────────────────────┼────────────────────────────────────────┤
│ Переписывает историю  │ Добавляет новый коммит-отмену          │
│ (SHA коммитов могут   │ (история не меняется)                  │
│  пропасть)            │                                        │
├───────────────────────┼────────────────────────────────────────┤
│ Локальные ветки,      │ Публичные ветки (уже pushed),          │
│ которые ещё не        │ общие ветки команды                    │
│ были pushed           │                                        │
├───────────────────────┼────────────────────────────────────────┤
│ «Перекроить историю   │ «Отменить изменение, сохранив          │
│  до push»             │  видимый след об отмене»               │
└───────────────────────┴────────────────────────────────────────┘
```

### Git Reflog — машина времени

Reflog записывает **все** перемещения HEAD, даже те, которые не создали коммита. Это ваша страховка от случайного `--hard reset`.

```bash
# Посмотреть reflog
git reflog

# Вывод:
# abc1234 HEAD@{0}: reset: moving to HEAD~1
# def5678 HEAD@{1}: commit: feat: добавлен factorial
# ghi9012 HEAD@{2}: checkout: moving from feature to main

# Восстановить «потерянный» коммит:
git checkout def5678      # проверить содержимое
git switch -c rescue/lost-commit   # создать ветку из него
# или:
git reset --hard def5678  # вернуть HEAD туда
```

### Удаление коммитов: полный обзор

```bash
# СЦЕНАРИЙ 1: Отменить последний коммит, сохранить изменения
git reset --soft HEAD~1

# СЦЕНАРИЙ 2: Убрать последний коммит полностью (локально)
git reset --hard HEAD~1

# СЦЕНАРИЙ 3: Отменить коммит, который уже был запушен
git revert HEAD
git push

# СЦЕНАРИЙ 4: Переписать историю последних N коммитов (до push)
git rebase -i HEAD~3
# В открывшемся редакторе:
# pick abc1234 feat: модуль greeter      ← оставить
# squash def5678 fix: опечатка           ← слить с предыдущим
# drop ghi9012 WIP: незаконченное        ← удалить

# СЦЕНАРИЙ 5: Исправить сообщение последнего коммита
git commit --amend -m "feat: правильное сообщение"
# Внимание: amend переписывает коммит (новый SHA)!

# СЦЕНАРИЙ 6: Добавить забытый файл в последний коммит
git add forgotten_file.cpp
git commit --amend --no-edit
```

## 8. Работа с удалённым репозиторием

### Основной цикл push/pull

```
┌─────────────────────────────────────────────────────────────┐
│                СИНХРОНИЗАЦИЯ С REMOTE                        │
│                                                             │
│  Локальный          Удалённый                               │
│  репозиторий        репозиторий (GitHub)                    │
│                                                             │
│  [main] A─B─C  ──── git push ────►  [origin/main] A─B─C    │
│                                                             │
│  [main] A─B─C  ◄─── git pull ────  [origin/main] A─B─C─D   │
│              \                                              │
│               D  (D применён локально)                      │
│                                                             │
│  git fetch: только скачивает, не применяет:                 │
│  [main] A─B─C       [origin/main] A─B─C─D                  │
│  (ваша ветка не тронута)                                    │
└─────────────────────────────────────────────────────────────┘
```

```bash
# Клонировать репозиторий
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git  # через SSH

# Посмотреть удалённые ветки
git branch -r

# Посмотреть все ветки (локальные + удалённые)
git branch -a

# Отправить изменения
git push origin main
# При первом push новой ветки:
git push -u origin feature/add-farewell
# (-u устанавливает upstream — потом просто git push)

# Получить изменения
git fetch origin          # скачать, не применять
git pull                  # fetch + merge
git pull --rebase         # fetch + rebase (линейная история)

# Удалить удалённую ветку
git push origin --delete feature/old-branch

# Обновить список удалённых веток (удалить устаревшие ссылки)
git fetch --prune
git remote prune origin
```

### Pull Request / Merge Request

```
ТИПИЧНЫЙ WORKFLOW С PR:

  1. Создать ветку от main:
     git switch -c feature/new-function

  2. Работать, делать коммиты:
     git add . && git commit -m "feat: ..."

  3. Запушить ветку:
     git push -u origin feature/new-function

  4. Открыть PR на GitHub:
     → Compare & pull request
     → Заполнить описание: что сделано, зачем, как тестировать

  5. Code review:
     → Коллеги смотрят diff, оставляют комментарии
     → Вы исправляете, пушите новые коммиты в ту же ветку
     → LGTM → Approve

  6. Merge PR (GitHub):
     → Squash and merge    (один коммит, чистая история)
     → Rebase and merge    (линейная история с исходными коммитами)
     → Create merge commit (стандартный merge-коммит)

  7. Удалить ветку:
     → На GitHub кнопка "Delete branch"
     → Локально: git branch -d feature/new-function
```

## 9. Монорепозиторий

### Что такое монорепозиторий

Монорепозиторий — это **один репозиторий**, содержащий несколько отдельных проектов или пакетов.

```
polytrepo (несколько репо):       monorepo (один репо):
  github.com/org/frontend            github.com/org/platform/
  github.com/org/backend               ├── frontend/
  github.com/org/mobile                ├── backend/
  github.com/org/shared-libs           ├── mobile/
                                       └── shared-libs/
```

### Плюсы и минусы монорепо

```
┌─────────────────────────────────────────────────────────────┐
│                  МОНОРЕПОЗИТОРИЙ                            │
├──────────────────────────┬──────────────────────────────────┤
│  ПЛЮСЫ                   │  МИНУСЫ                          │
├──────────────────────────┼──────────────────────────────────┤
│ ✓ Атомарные коммиты      │ ✗ Размер репозитория растёт      │
│   (изменения в frontend  │ ✗ git clone медленнее            │
│   и backend в 1 коммите) │ ✗ CI/CD сложнее настроить        │
│ ✓ Единые инструменты     │ ✗ Права доступа сложнее          │
│   (linting, тесты)       │   (у всех доступ ко всему)       │
│ ✓ Легко рефакторить      │ ✗ Нужны инструменты для          │
│   общий код              │   работы с монорепо              │
│ ✓ Одна версия            │   (nx, turborepo, bazel)         │
│   зависимостей           │                                  │
├──────────────────────────┴──────────────────────────────────┤
│  Используют: Google, Meta, Microsoft, Uber, Twitter         │
└─────────────────────────────────────────────────────────────┘
```

### Инструменты для монорепо (C++ и не только)

```
Универсальные:
  • Bazel (Google)  — мощный, сложный
  • CMake           — стандарт для C++
  • nx              — JavaScript/TypeScript
  • turborepo       — JavaScript/TypeScript

Для нашего C++ учебного монорепо используем CMake:
```

### Пример учебного монорепо (C++)

```bash
# Структура нашего учебного монорепо:
cpp-monorepo/
├── CMakeLists.txt          ← корневой CMake
├── .gitignore
├── README.md
├── libs/
│   ├── greeter/
│   │   ├── CMakeLists.txt
│   │   ├── greeter.h
```

```cmake

add_subdirectory(libs/greeter)

add_subdirectory(apps/hello)
```

## Шпаргалка — самые нужные команды

```bash
# ─── ИНИЦИАЛИЗАЦИЯ ───────────────────────────────────────────
git init                     # новый репозиторий
git clone <url>              # клонировать существующий

# ─── БАЗОВЫЙ ЦИКЛ ────────────────────────────────────────────
git status                   # что изменилось?
git add <file>               # добавить в staging
git add .                    # добавить всё
git commit -m "message"      # зафиксировать
git log --oneline            # история

# ─── ВЕТКИ ───────────────────────────────────────────────────
git branch                   # список веток
git switch -c <name>         # создать и переключиться
git switch <name>            # переключиться
git merge <branch>           # слить в текущую
git branch -d <name>         # удалить ветку
git rebase <branch>          # перебазировать

# ─── STASH ───────────────────────────────────────────────────
git stash push -m "desc"     # сохранить
git stash list               # список
git stash pop                # восстановить и удалить
git stash apply stash@{N}    # восстановить (сохраняя)
git stash drop stash@{N}     # удалить

# ─── ИСТОРИЯ ─────────────────────────────────────────────────
git diff                     # что изменилось (не staged)
git diff --staged            # что в staging
git log --graph --oneline    # красивое дерево
git show <SHA>               # детали коммита
git blame <file>             # кто писал каждую строку

# ─── ОТМЕНА ──────────────────────────────────────────────────
git restore <file>           # отменить изменения в файле
git restore --staged <file>  # убрать из staging
git reset --soft HEAD~1      # отменить коммит (staged)
git reset HEAD~1             # отменить коммит (unstaged)
git reset --hard HEAD~1      # отменить коммит (удалить)
git revert <SHA>             # безопасная отмена (новый коммит)
git commit --amend           # исправить последний коммит
git reflog                   # журнал всех перемещений HEAD

# ─── REMOTE ──────────────────────────────────────────────────
git remote -v                # список remote
git push -u origin <branch>  # запушить (первый раз)
git push                     # запушить
git fetch                    # скачать без применения
git pull                     # скачать и применить
git pull --rebase            # скачать и rebase
```
